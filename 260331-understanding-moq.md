# 低遅延・大規模ライブ配信を実現する Media over QUIC の気持ちを理解する

## 今日のテーマ

- 次世代のライブ配信技術 MoQ を紹介する
- 既存技術をおさらいしつつ、MoQ がなぜこう設計されたのか、その気持ちを理解する
- 「低遅延・大規模配信の世界おもしろいな、もうちょっと知りたいな」と思ってもらう

## HLS/DASH — 今のライブ配信の仕組み

- アニメーションで HLS/DASH の仕組みと遅延構造を説明する
  - 配信者がサーバーに映像を送る
  - サーバーが映像をセグメント (数秒の塊) に分割する
  - サーバーが Playlist にセグメントの URL を書き出す
  - 視聴者が Playlist を HTTP GET する
  - Playlist に書かれたセグメントを HTTP GET して再生する
  - これを繰り返す
  - 口頭で「セグメント完成待ち」「ポーリング待ち」「全部 HTTP (視聴者側)」を補足する

## LL-HLS — 低遅延化の工夫

- LL-HLS は HLS の遅延を 2 つの工夫で縮めた
  - Partial Segment: 親セグメント (2秒) の完成を待たずに、その一部 (0.5秒程度) を先に配信する。配信の単位を小さくすることで完成待ちの時間を短くした
  - Blocking Playlist Reload + Preload Hint: ポーリングをやめて、「次の Partial Segment ができたら教えて」「次の Partial Segment を先にリクエストしておく」という仕組みでポーリング待ちを解消
- これにより遅延は数秒〜数十秒から 2-3 秒程度に縮まった
- ただし HTTP のリクエスト/レスポンスの構造は変わっていない
- アニメーションで LL-HLS の仕組みを HLS との対比で説明する
  - 配信者がサーバーに映像を送る (HLS と同じ)
  - サーバーが映像を Partial Segment に分割して Playlist に書き出す。末尾に次の Partial Segment のヒントも書く
  - 視聴者が Playlist を GET する (初回は普通に)
  - Playlist にある Partial Segment を GET して再生する
  - 同時にヒントのリソースを先にリクエストする (サーバーはできたら返す)
  - 同時に Blocking Playlist Reload で「次の更新ができたら教えて」とリクエストする
  - サーバーが新しい Partial Segment を作る
  - Playlist に新しい Partial Segment の URL と次のヒントを書き出す
  - ヒントのレスポンスと Blocking Playlist Reload のレスポンスが返る
  - 視聴者はまた新しいヒントを先にリクエストし、Blocking Playlist Reload も送る
  - これを繰り返す
  - 口頭で「ポーリング待ちは解消した」「完成待ちは短くなった」「でも HTTP のリクエスト/レスポンスの構造は残っている」を補足する

## WebRTC — 低遅延だがスケールしない

- 静止画で WebRTC の低遅延を説明する
  - 配信者 → SFU → 視聴者 がリアルタイムに繋がっている
  - 口頭で「低遅延だが、スケールが難しい。HLS/DASH は HTTP だから CDN でスケールできたが、WebRTC にはそういうものがない。低遅延とスケーラビリティを両立するものがなかった」

## MoQ — 低遅延とスケーラビリティの両立

- 静止画でプロトコルスタックを見せる
  - MSF (映像・音声の乗せ方) / MoQT (Pub/Sub + データモデル) / QUIC (トランスポート) の 3 層
  - 「HTTP をやめて、QUIC の上にライブ配信のための新しいプロトコル MoQ を作った」
  - 口頭で「MoQ はこの 3 層で構成される。今日は主に MoQT と QUIC の話をします。この先、どうやって低遅延とスケーラビリティを実現するか見ていく」

## MoQ の低遅延 — フレーム単位の配信

- アニメーションで MoQ の低遅延を HLS/LL-HLS との対比で説明する
  - 配信者が Relay に映像をフレーム単位 (Object) で送っている
  - 視聴者が Relay に SUBSCRIBE する
  - Relay が Object を視聴者に転送する
  - 視聴者が Object を受け取って再生する
  - これが繰り返される
  - 口頭で「セグメント完成待ちもポーリングもない。SUBSCRIBE したら Object が順次届く」
  - 口頭で「本当はセッション確立や Namespace の発見などのステップがあるが省略している」

## MoQ のスケーラビリティ — Relay による配信のスケール

- アニメーションで Relay によるスケーラビリティを説明する
  - 低遅延セクションのアニメーションの続き (配信者 → Relay → 視聴者)
  - 視聴者が増えていく → Relay が fan-out して配る
  - さらに視聴者が増える → Relay を多段に配置してスケール
  - 口頭で「Relay を多段に配置するだけでスケールできる」

## まとめ

- HLS/DASH はセグメント + HTTP GET の仕組みで、構造的に遅延がある
- LL-HLS は改善したが、HTTP のリクエスト/レスポンスの枠内
- WebRTC は低遅延だが、スケールしない
- MoQ はこれらの課題を踏まえて、QUIC の上でライブ配信プロトコルを新たに設計した
  - SUBSCRIBE でフレーム単位の Object が順次流れてくる → 低遅延
  - Relay の Pub/Sub で fan-out + 多段配置 → スケーラビリティ
- 映像・音声のライブ配信技術って面白そうだなと思ってもらえたら嬉しいです
